I INTRODUCTION** --- ### 1. Basics of Cryptography * **Definition**: Science of securing information through transformation (encryption) * **Goals**: * **Confidentiality**: Prevent unauthorized disclosure * **Integrity**: Ensure data is unaltered * **Authentication**: Verify sender/receiver identities * **Non-repudiation**: Prevent denial of actions * **Components**: * **Plaintext**: Original readable data * **Ciphertext**: Encrypted data * **Key**: Secret parameter used for encryption/decryption * **Encryption Algorithm**: Method for converting plaintext to ciphertext * **Decryption Algorithm**: Method for recovering plaintext from ciphertext * **Real-World Applications**: Secure messaging (WhatsApp), online banking (SSL/TLS), data storage (AES encryption) * **diagram: false** --- ### 2. Conventional (Symmetric) Cryptography * **Definition**: Single secret key used for both encryption and decryption * **Algorithms**: * **DES (Data Encryption Standard)**: 56-bit key, outdated * **AES (Advanced Encryption Standard)**: 128/192/256-bit keys, current standard * **Strengths**: Fast for large data volumes; low computational overhead * **Weaknesses**: Secure key distribution required; single point of compromise if key exposed * **diagram: true** * Search: `symmetric encryption process diagram` --- ### 3. Public-Key (Asymmetric) Cryptography * **Definition**: Two mathematically linked keys (public and private) * **Key Pair**: * **Public Key**: Shared openly for encryption or signature verification * **Private Key**: Kept secret for decryption or signature creation * **Algorithms**: * **RSA**: Based on factoring large primes * **ECC (Elliptic Curve Cryptography)**: Smaller keys, faster operations * **Use Cases**: Secure key exchange, digital signatures, SSL/TLS handshake * **Strengths**: Solves key distribution; enables digital signatures * **Weaknesses**: Slower than symmetric; computationally intensive * **diagram: true** * Search: `public key cryptography workflow diagram` --- ### 4. Hash Functions * **Definition**: One-way function mapping arbitrary-length input to fixed-length digest * **Properties**: * **Deterministic**: Same input → same output * **One-Way (Preimage Resistance)**: Cannot invert digest to original message * **Collision Resistance**: Hard to find two inputs with same hash * **Quick Computation**: Efficient to compute for any input * **Algorithms**: MD5 (obsolete), SHA-1 (weak), SHA-256/SHA-3 (current standard) * **Applications**: Password storage (salting + hashing), data integrity checks, digital signatures * **diagram: false** --- ### 5. Authentication * **Definition**: Process of verifying identity of a user/system * **Factors of Authentication**: * **Something You Know**: Password, PIN * **Something You Have**: Token, smart card * **Something You Are**: Biometrics (fingerprint, iris) * **Types**: * **Single-Factor**: Password only * **Two-Factor (2FA)**: Password + OTP/token * **Multi-Factor**: Any combination of factors * **Mechanisms**: Challenge–response protocols, one-time passwords, digital certificates * **diagram: false** --- ### 6. Digital Signatures * **Definition**: Cryptographic scheme ensuring data authenticity and integrity * **Process**: 1. **Hash Message**: Compute digest of message using hash function 2. **Encrypt Digest**: Encrypt digest with sender’s private key → signature 3. **Transmit**: Send message + signature 4. **Verify**: Receiver decrypts signature with sender’s public key → obtains digest; hashes received message; compares digests * **Security Services**: * **Authentication**: Verifies sender’s identity * **Integrity**: Detects message tampering * **Non-repudiation**: Prevents sender denial * **diagram: true** * Search: `digital signature generation and verification diagram` --- **UNIT II KEY MANAGEMENT AND AUTHENTICATION** --- ### 1. Symmetric Key Distribution * **Requirement**: Securely share secret key between parties before communication * **Methods**: * **Manual Exchange**: Physical transfer (e.g., USB, paper) * **Key Distribution Center (KDC)**: Trusted third party (e.g., Kerberos KDC) * **Diffie–Hellman Key Exchange**: Establish shared secret over insecure channel without transmitting key * **Challenges**: Eavesdropping risk, Scalability (O(n²) for n users), Trust establishment * **diagram: true** * Search: `diffie hellman key exchange diagram` --- ### 2. Distribution of Public Keys * **Requirement**: Ensure authenticity of public keys to prevent man-in-the-middle * **Methods**: * **Public Key Directory**: Centralized repository of public keys * **Public-Key Certificates (X.509)**: CA-signed documents binding identity to key * **Web of Trust (PGP)**: Users sign each other’s public keys * **diagram: false** --- ### 3. X.509 Certificates * **Structure**: * **Version** (v3) * **Serial Number** * **Signature Algorithm** * **Issuer** (CA name) * **Validity Period** (`Not Before`, `Not After`) * **Subject** (entity’s name) * **Subject Public Key Info** (algorithm, key) * **Extensions** (Key Usage, CRL Distribution Points, Subject Alternative Name) * **Signature** (CA’s private key signature) * **Usage**: SSL/TLS, S/MIME, code signing, VPN authentication * **Advantages over Traditional Methods**: Scalability, revocation via CRL/OCSP, tamper resistance * **diagram: true** * Search: `x.509 certificate structure diagram` --- ### 4. Public-Key Infrastructure (PKI) * **Components**: * **Certificate Authority (CA)**: Issues and revokes certificates * **Registration Authority (RA)**: Verifies identity before certificate issuance * **Certificate Repository**: Stores issued certificates and CRLs * **Certificate Revocation List (CRL) / OCSP**: Lists revoked certificates * **Users/Devices**: Entities possessing key pairs and certificates * **Functions**: * **Certificate Issuance**: CA signs user’s public key → certificate * **Certificate Management**: Renewal, revocation, distribution * **Trust Model**: Hierarchical (root CA → intermediate CAs → end-entity) * **Benefits**: Scalable, automated public key verification, non-repudiation, secure email (S/MIME), SSL/TLS * **diagram: true** * Search: `public key infrastructure PKI diagram` --- ### 5. Remote User-Authentication Principles * **Goals**: Verify user identity over insecure network without revealing secrets, prevent replay, resist MITM * **Concepts**: * **Challenge–Response**: Server sends random nonce; client encrypts nonce with shared secret or private key * **Use of Nonces/Timestamps**: Prevent replay attacks * **Session Keys**: Temporary keys for ensuing communication * **diagram: false** --- ### 6. Remote User-Authentication Using Symmetric Encryption * **Process (Simplified)**: 1. **Client** → “I want to authenticate” 2. **Server** → Sends random nonce 3. **Client** → Encrypts nonce with shared secret key, returns ciphertext 4. **Server** → Decrypts with shared key, verifies nonce → Authentication success * **Weaknesses**: Requires pre-shared key; no non-repudiation; vulnerable if key compromised * **diagram: true** * Search: `symmetric challenge response authentication diagram` --- ### 7. Kerberos Systems * **Architecture**: * **Authentication Server (AS)** * **Ticket-Granting Server (TGS)** * **Service Server (SS)** * **Client** (user) * **Workflow**: 1. **Client → AS**: “Here’s my username” 2. **AS → Client**: Ticket-Granting Ticket (TGT) encrypted with client’s key 3. **Client**: Decrypts TGT, stores it 4. **Client → TGS**: Presents TGT, requests service ticket for SS 5. **TGS → Client**: Service ticket encrypted with SS’s key 6. **Client → SS**: Presents service ticket → Access granted * **Security Features**: Single Sign-On (SSO), mutual authentication, time-limited tickets (prevents replay), reliance on symmetric keys * **diagram: true** * Search: `kerberos authentication flow diagram` --- ### 8. Remote User-Authentication Using Asymmetric Encryption * **Process**: 1. **Client** → “I want to authenticate” 2. **Server** → Sends challenge (random nonce) 3. **Client** → Signs nonce with its private key, attaches certificate 4. **Client** → Sends signed nonce + certificate to server 5. **Server** → Verifies certificate (via CA’s public key), uses client’s public key to verify signature → Authentication success * **Advantages**: No pre-shared secret; strong non-repudiation; resistant to replay if nonce/timestamp used * **diagram: true** * Search: `asymmetric challenge response authentication diagram` --- **UNIT III ACCESS CONTROL AND SECURITY** --- ### 1. Network Access Control (NAC) * **Definition**: Mechanism to restrict network access only to authorized devices/users * **Components**: * **Supplicant**: Client device requesting access * **Authenticator**: Network device (switch or AP) enforcing policy * **Authentication Server**: Typically RADIUS server verifying credentials * **Process Flow (802.1X)**: 1. **Supplicant → Authenticator**: “Let me in” 2. **Authenticator**: Blocks all traffic except EAP 3. **Supplicant → Authenticator**: Sends EAP credentials 4. **Authenticator → RADIUS**: Forwards credentials 5. **RADIUS → Authenticator**: Accept/Deny 6. **Authenticator**: Opens port for accepted supplicant; denies others * **Benefits**: Prevents unauthorized access to LAN or Wi-Fi; enforces endpoint security checks * **diagram: true** * Search: `IEEE 802.1X authentication process diagram` --- ### 2. IP Security (IPsec) * **Purpose**: Provide secure IP-level communication (confidentiality, integrity, authentication) * **Protocols**: * **AH (Authentication Header)**: Provides integrity and authentication, no encryption * **ESP (Encapsulating Security Payload)**: Provides encryption, integrity, and authentication * **Modes**: * **Transport Mode**: Encrypts only payload, used for host-to-host communication * **Tunnel Mode**: Encrypts entire IP packet, encapsulates in new IP header, used for network-to-network or VPN * **diagram: true** * Search: `IPsec transport mode vs tunnel mode diagram` --- ### 5. Internet Key Exchange (IKE) * **Role**: Establishes Security Associations (SAs) and shared keys for IPsec * **Phases**: 1. **IKE Phase 1**: * Establishes **IKE SA** (secure channel) * Authenticates peers (pre-shared key or certificates) * Performs Diffie–Hellman exchange → Generates shared secret 2. **IKE Phase 2**: * Uses IKE SA to negotiate **IPsec SAs** for data encryption * Negotiates encryption algorithms, keys, lifetimes * **Versions**: * **IKEv1**: Two-step message exchange, more complex * **IKEv2**: Streamlined four-message exchange, built-in NAT traversal, improved DoS resistance * **diagram: true** * Search: `IKE phase 1 and phase 2 diagram IPsec` --- ### 6. Web Security Considerations * **Common Threats**: * **XSS (Cross-Site Scripting)**: Injecting malicious scripts into web pages * **SQL Injection**: Inserting malicious SQL to manipulate databases * **CSRF (Cross-Site Request Forgery)**: Forcing user to perform actions without consent * **Session Hijacking**: Stealing session tokens or cookies * **Countermeasures**: * Input validation and output encoding (prevent XSS) * Parameterized queries (prevent SQL injection) * CSRF tokens (prevent CSRF) * Secure cookie flags (`HttpOnly`, `Secure`) * Use HTTPS to prevent eavesdropping * **diagram: false** --- ### 7. Secure Sockets Layer (SSL) * **Purpose**: Encrypt data in transit between client and server at transport layer; predecessor of TLS * **Handshake Steps**: 1. **ClientHello**: Client sends supported SSL version, cipher suites, random number 2. **ServerHello**: Server chooses protocol, sends digital certificate (X.509), random number 3. **Certificate Verification**: Client validates server certificate via CA public key 4. **Pre-Master Key**: Client generates a pre-master key, encrypts it with server’s public key, sends it 5. **Session Key Generation**: Both derive symmetric session key from pre-master and random numbers 6. **ChangeCipherSpec**: Both switch to encrypted mode 7. **Finished**: Exchange final encrypted messages → Secure channel established * **Limitations**: SSLv2/SSLv3 deprecated due to vulnerabilities (POODLE, BEAST) * **diagram: true** * Search: `SSL handshake diagram step by step` --- ### 8. Transport Layer Security (TLS) * **Successor of SSL**; current versions: TLS 1.2, TLS 1.3 * **Improvements over SSL**: * Reduced handshake steps in TLS 1.3 (faster) * Stronger cipher suites (AES-GCM, ChaCha20-Poly1305) * Built-in support for forward secrecy * **Use Cases**: HTTPS, secure email (SMTPS, IMAPS), VPNs * **diagram: false** (covered by SSL handshake with TLS substitution) --- ### 9. Pretty Good Privacy (PGP) * **Purpose**: Secure email and file encryption using hybrid cryptography * **Process**: 1. **Generate Session Key**: Random symmetric key (e.g., AES) to encrypt email body 2. **Encrypt Message**: Use session key to encrypt data (faster than asymmetric for bulk) 3. **Encrypt Session Key**: Use recipient’s public key to encrypt the session key 4. **Digital Signature**: Hash plaintext, encrypt hash with sender’s private key, attach to message 5. **Transmit**: Send encrypted session key + encrypted message + signature 6. **Recipient**: Decrypt session key with private key, decrypt message, verify signature with sender’s public key * **Security Services**: Confidentiality, Authentication, Integrity, Non-repudiation * **diagram: true** * Search: `PGP encryption and decryption flow diagram` --- ### 2. S/MIME (Secure/Multipurpose Internet Mail Extensions) * **Purpose**: Standard protocol for securing MIME-based email using certificates * **Components**: * **X.509 Certificates**: Issued by CA, bind user identity to public key * **CMS (Cryptographic Message Syntax)**: Defines structure for signed/encrypted messages * **Process**: 1. **Digital Signing**: Hash message, encrypt hash with sender’s private key → Signature 2. **Encryption**: Encrypt message+signature with recipient’s public key 3. **Transmission**: Send encrypted, signed email 4. **Decryption**: Recipient decrypts with private key, verifies signature with sender’s public key * **Applications**: Corporate email security, legal document exchange * **diagram: true** * Search: `S/MIME email encryption and signature diagram` --- ### 3. DomainKeys Identified Mail (DKIM) * **Purpose**: Authenticate email sender’s domain and ensure message integrity * **Mechanism**: 1. **Generate Hash**: Sender computes hash over selected headers + body 2. **Sign Hash**: Encrypt hash with domain’s private key → DKIM signature placed in email header 3. **Publish Public Key**: Stored in DNS TXT record under `selector._domainkey.example.com` 4. **Verification**: Receiver retrieves public key from DNS, decrypts signature, compares hash to recomputed value → Valid if match * **Benefits**: Prevents email spoofing, improves deliverability, works with SPF/DMARC * **diagram: true** * Search: `DKIM email authentication process diagram` --- ### 4. Wireless Network Security: Mobile Device Security * **Challenges**: * Unsecured public Wi-Fi (eavesdropping) * Lost/stolen devices (data leakage) * Malware-infected apps (spyware, ransomware) * OS vulnerabilities (delayed patches) * BYOD (Bring Your Own Device) policies (diverse device security) * **Security Measures**: * **Device Encryption**: Full-disk encryption or file encryption (Android Keystore, iOS Data Protection) * **Mobile Device Management (MDM)**: Enforce policies, remote wipe, app blacklisting (e.g., Microsoft Intune, AirWatch) * **Strong Authentication**: Biometric (fingerprint, face ID), PIN/pattern * **Secure Boot & Trusted Execution Environment (TEE)**: Verify OS integrity at startup * **VPN Usage**: Encrypt traffic on public networks * **Application Sandboxing**: Limit app access to system resources * **diagram: false** --- **UNIT V SECURITY PRACTICES** --- ### 1. Intrusion Detection Systems (IDS) * **Definition**: Systems that monitor network or host activities to detect malicious actions or policy violations * **Types**: * **Host-Based IDS (HIDS)**: Monitors individual hosts (file integrity checks, log analysis) * **Network-Based IDS (NIDS)**: Monitors network traffic at strategic points (packet capture, protocol analysis) * **Detection Approaches**: * **Signature-Based**: Matches traffic patterns against database of known attack signatures (fast, low false positives but cannot detect new attacks) * **Anomaly-Based**: Establishes baseline of normal behavior, flags deviations (can detect zero-day but higher false positives) * **Components**: * **Sensors/Agents**: Collect data (logs, packets) * **Analysis Engine**: Compares data to rules or profiles * **Alerting System**: Notifies administrators on detection * **Integration with IPS**: Intrusion Prevention Systems can block detected attacks in real time * **diagram: true** * Search: `intrusion detection system architecture diagram` --- ### 2. Password Management * **Principles**: * **Password Complexity**: Minimum length, mix of uppercase, lowercase, numbers, special chars * **Password Storage**: Store hashed (salt + hash, e.g., bcrypt, Argon2), never plaintext * **Password Policies**: Expiration, reuse prevention, account lockout after failed attempts * **Multi-Factor Authentication (MFA)**: Combine password with OTP, token, or biometric * **Threats**: * **Brute-Force & Dictionary Attacks**: Automated guessing of common passwords * **Phishing & Social Engineering**: Trick users into revealing credentials * **Keylogging**: Malware capturing keystrokes * **diagram: false** --- ### 3. Firewall Characteristics and Types of Firewalls * **Characteristics**: * **Traffic Filtering**: Block/allow based on IP address, port, protocol * **Stateful Inspection**: Track active connections, context-aware filtering * **Application Layer Filtering (Proxy)**: Inspect application data (HTTP, SMTP) * **Default Deny vs Default Allow**: Secure posture typically “deny by default” * **Logging & Alerting**: Record suspicious events, send alerts for anomalies * **Types of Firewalls**: 1. **Packet-Filtering Firewall**: Inspects packet headers (fast, limited context) 2. **Stateful Inspection Firewall**: Tracks connection states (more secure, moderate performance) 3. **Proxy (Application Layer) Firewall**: Intermediary for application traffic (deep security, higher latency) 4. **Next-Generation Firewall (NGFW)**: Combines firewall, IDS/IPS, DPI, application awareness (comprehensive, resource-intensive) * **diagram: false** --- ### 4. Firewall Basing, Location, and Configurations * **Firewall Basing**: Deploys firewall at network boundaries, DMZ, and internal segments for layered defense * **Firewall Locations**: * **Perimeter (Edge) Firewall**: Between Internet and DMZ or LAN * **DMZ Firewall**: Between DMZ and internal network * **Internal Segment Firewall**: Between departments or VLANs inside LAN * **Basic Configuration Steps**: 1. **Define Security Policy**: Determine allowed/blocked traffic (port, IP, protocol) 2. **Implement Access Control Lists (ACLs)**: Deny all by default; allow only necessary services (e.g., HTTP, HTTPS to web server) 3. **Configure NAT/PAT**: Translate public IPs to private IPs for internal hosts 4. **Enable Logging & Alerts**: Monitor denied attempts, port scans, or anomalies 5. **Regular Rule Review**: Update rules based on new threats or business requirements * **diagram: true** * Search: `corporate network firewall DMZ internal LAN diagram` --- ### 5. Blockchains * **Definition**: Distributed, immutable ledger of transactions grouped in blocks, chained via cryptographic hashes * **Key Concepts**: * **Decentralization**: No single controlling entity; each node holds a copy of the ledger * **Consensus Mechanisms**: Proof of Work (PoW), Proof of Stake (PoS), or Practical Byzantine Fault Tolerance (PBFT) to validate blocks * **Immutability**: Each block contains previous block’s hash → tampering changes hashes downstream * **Smart Contracts**: Self-executing code stored on blockchain for automated trust * **Applications for Data Integrity**: Supply chain tracking, financial transactions, digital identity, voting systems * **diagram: true** * Search: `blockchain data integrity architecture diagram` --- ### 6. Cloud Security * **Challenges**: * **Multi-Tenancy Risks**: Multiple users share physical resources (isolation concerns) * **Data Breaches**: Unauthorized access to stored data (misconfigured S3 buckets, weak access controls) * **Insecure APIs**: Weak authentication, excessive privileges * **Insider Threats**: Employees or third-party providers with privileged access * **Security Measures**: * **Encryption at Rest & In Transit** (AES, TLS) * **Identity and Access Management (IAM)**: Role-based access control (RBAC), least privilege * **Security Monitoring and Logging**: SIEM, CloudTrail for AWS, Azure Monitor * **Network Security**: Virtual private clouds (VPCs), security groups, firewall rules * **Compliance & Governance**: Adhere to standards (ISO 27001, GDPR, HIPAA) * **diagram: false** --- ### 7. IoT Security * **Challenges**: * **Resource Constraints**: Limited CPU, memory, making strong crypto difficult * **Heterogeneous Devices**: Diverse OS, protocols, interoperability issues * **Physical Access**: Devices in public places can be tampered with * **Weak Authentication**: Default or hardcoded credentials in devices * **Firmware Vulnerabilities**: Unpatched outdated firmware leads to exploits * **Security Measures**: * **Device Authentication**: Unique device certificates, secure boot * **Data Encryption**: Lightweight cryptography (Elliptic Curve, AES-CCM) for data in transit and at rest * **Secure Firmware Updates**: Signed firmware images, over-the-air (OTA) update security * **Network Segmentation**: Isolate IoT devices in separate VLANs or subnets, use firewalls at IoT gateways * **Monitoring & Anomaly Detection**: Behavioral analytics, intrusion detection on IoT network * **diagram: false**
